# Философия и принципы: Подстрочник

> Если показать это расширение Сергею Брину — он поставит его в музей Google
> как идеальный Chrome Extension. Permissions минимальны, код чист,
> UX безупречен, безопасность без компромиссов.

---

## Миссия

Я люблю слушать оригинальные голоса — акценты, интонации, как звучат разные языки.
Не люблю дубляж. Но русских субтитров часто нет: редкие сериалы, свежие сезоны,
или студия считает, что дубляж и так всё закрывает.

Подстрочник — инструмент для тех, кто хочет слышать оригинал и понимать каждое слово.
Не замена профессионального перевода, а альтернатива дубляжу.

---

## Принцип Брина (120%)

> Расширение, которое покажут в музее Google как идеал Chrome Extension.

**Как проверить:**
- Permissions — минимально необходимые, каждое обосновано
- UX — зритель не знает что такое HLS, API-ключ, батч. И не должен
- Код — идиоматичный MV3, как из документации Chrome Extensions
- Безопасность — ни один reviewer в CWS не найдёт замечаний
- Производительность — расширение не замедляет страницу

## Принципы продукта

| # | Принцип | Суть |
|---|---------|------|
| 1 | **Зритель в центре** | Нажал Play — читает перевод. Каждый лишний шаг — провал |
| 2 | **Качество > покрытие** | Один сериал с Opus лучше десяти с плохим переводом. Лучшая модель побеждает |
| 3 | **Работает без инфраструктуры** | Extension-first. API-ключ + расширение = всё. Сервер, сайт — усиление, не зависимость |
| 4 | **Прозрачность** | Код открыт, данные не собираются, расширение бесплатно. Ошибки видны, не спрятаны |
| 5 | **Кеш — общее благо** | Перевод делается один раз. Shared cache = каждый следующий зритель получает мгновенно |

## Три зоны качества

| Зона | Стандарт | Почему |
|------|----------|--------|
| **Спека** | Перфекционизм | Переделать спеку в 10× дешевле чем код. Ошибка в архитектуре дорогая |
| **Код** | Прагматизм | Работающий код лучше идеального ненаписанного. Агент может написать неидеально — ок |
| **Безопасность** | Паранойя | Публичный VPS, API-ключи пользователей, расширение на чужих сайтах. Нет «и так сойдёт» |

## Zen of Подстрочник

### Код

- Background делает всё. Content script — только глаза и руки.
- Один файл лучше двух. Ноль зависимостей лучше одной.
- Если агент не понял код за 30 секунд — код слишком сложный.
- Кеш — не оптимизация, а продукт. Без кеша нет Подстрочника.
- Ошибка, которую видит пользователь, лучше ошибки, которую не видит никто.
- Не пиши код без спеки. Переделать спеку в 10 раз дешевле.

### Дизайн

- Субтитры незаметны. Расширение тоже.
- Если нужна инструкция — дизайн провалился.
- Один клик лучше двух. Ноль кликов лучше одного.
- Зритель не знает что такое HLS, API-ключ, батч. И не должен.

### Безопасность

- API-ключ не покидает устройство. Точка.
- `textContent`, не `innerHTML`. Всегда.
- Всё что с чужого сайта — враждебно до доказательства обратного.
- Localhost ≠ безопасно. `127.0.0.1`, не `0.0.0.0`.

### SEO

- Если Google не видит — этого не существует.
- Статический HTML > JS-рендеринг. Всегда.
- Каждая страница — отдельная точка входа с собственным title.
- URL читаем человеком. `/ru/subtitles/sherlock`, не `/page?id=37`.

### Архитектура

- Extension-first. Сервер, сайт, VPS — усиление, не зависимость.
- Спека перфектна. Код прагматичен. Безопасность параноидальна.
- Данные ≠ инструкции. Субтитры в промпте — данные, не команды.
- Лучшая модель побеждает. `model_rank` — закон, не рекомендация.

### Мультиязычность

- Русский — первый, не единственный. Архитектура готова к `/en/` с первого дня.
- Добавить язык = добавить JSON. Не трогать шаблоны, не трогать код.
- `hreflang` — не когда-нибудь, а с первой страницы.
- URL говорит на каком языке: `/ru/subtitles/`, `/en/subtitles/`. Без параметров, без кук.

### Размер кода

- Файл ≤ 500 строк. Больше — разбить или обосновать в комментарии.
- Функция ≤ 50 строк. Если длиннее — она делает слишком много.
- Вложенность ≤ 3 уровня. `if` внутри `if` внутри `for` — рефакторить.
- Один файл = одна ответственность. `background.js` = движок перевода, не UI.

> Лимит мягче чем в Amsterdam (500 vs 200) — vanilla JS без модульной системы,
> import/export нет. Но принцип тот же: агент должен держать весь файл в голове.

### Вес сайта

- Страница ≤ 100 KB без кеша (HTML + CSS + JS). Без фреймворков это легко.
- Шрифты — Google Fonts CDN (кешируются глобально). Не бандлить.
- Изображения — SVG где можно, WebP где нельзя. Ноль растра на лендинге.
- Zero JS для рендера контента. JS — только интерактивность (accordion, scroll reveal).
- Core Web Vitals: LCP < 2.5s, CLS = 0, FID < 100ms. Статика через Nginx — почти бесплатно.

```
│   ├── providers.js    # Shared: провайдеры, VTT-парсер, промпт
│   ├── background.js   # Движок: перевод, кеш, API
│   ├── content.js      # UI: picker, overlay, рендер
│   └── popup.html      # Настройки
├── server/
│   ├── shared_cache.py # VPS: API + SQLite + очередь
│   └── server.py       # Воркер: polling + Claude CLI
├── site/
│   ├── templates/      # Jinja2-шаблоны
│   ├── static/         # CSS, JS
│   ├── i18n/           # Языковые файлы (JSON)
│   ├── generate.py     # SQLite → HTML
│   └── out/            # Сгенерированный сайт (gitignore)
├── docs/
│   ├── ARCHITECTURE.md # Архитектурные решения
│   ├── CONTRIBUTING.md # Гайд для контрибьюторов
│   └── internal/       # TODO.md, спеки, аудиты
├── PHILOSOPHY.md       # ← этот документ
├── CLAUDE.md           # Инструкции для AI-агента
├── CHANGELOG.md        # Keep a Changelog
└── README.md           # Для пользователей
```

### Границы компонентов

| Компонент | Знает о | Не знает о |
|-----------|---------|------------|
| **extension/** | Chrome API, OpenRouter API, VPS API | SQLite, Jinja2, Nginx |
| **server/** | SQLite, Claude CLI, VPS | Chrome API, UI, шаблоны |
| **site/** | SQLite (read-only), шаблоны, i18n | Chrome API, перевод, очередь |

- Extension и server общаются только через HTTP API (VPS).
- Site читает ту же БД что server, но только на чтение.
- Ни один компонент не импортирует файлы другого.

## Углы оценки спеки (7 штук)

Каждая спека перед реализацией оценивается по 7 углам. Шкала 1–10.

**Правило:** максимум 2 угла могут получить 9. Остальные — только 10. Ниже 9 — переписываем.

| # | Угол | Что проверяет | Критерии 10/10 |
|---|------|---------------|----------------|
| 1 | **Ценность** | Зачем это зрителю/переводчику? | Решает реальную проблему. Улучшение ощутимо без объяснений. Стоит усилий |
| 2 | **Архитектура** | Как вписывается в систему? | Следует принципам (extension-first, background proxy). Минимум новых связей. Не создаёт tech debt |
| 3 | **Полнота** | Всё ли описано? | Все сценарии (happy path + edge cases). Data flow от начала до конца. Нет «а что если...» |
| 4 | **Простота** | Минимум ли это? | Можно объяснить за 30 секунд. Нет over-engineering. Агент реализует за одну сессию |
| 5 | **Устойчивость** | Что если сломается? | API недоступен → понятное сообщение. Данных нет → fallback. Невалидный ввод → отбой без краша |
| 6 | **Безопасность** | Что может утечь? | Ключи не утекают. textContent. Rate limit. Публичные эндпоинты ≠ приватные данные |
| 7 | **Проверяемость** | Как узнать что готово? | Конкретные шаги (не «проверить»). Проверка за 5 минут. Критерии — done/not done |

### Формат оценки в спеке

```markdown

| Угол | v1 | v2 | Комментарий |
|------|:--:|:--:|-------------|
| Ценность | 10 | — | ... |
| Архитектура | 8 | 10 | Добавлен data flow |
| Полнота | 9 | — | Допустимо: edge case X маловероятен |
| Простота | 10 | — | ... |
| Устойчивость | 7 | 10 | Добавлен fallback при пустой БД |
| Безопасность | 10 | — | ... |
| Проверяемость | 9 | — | Допустимо: визуальная проверка |
| **Итого 9-ок** | **2** | **1** | **≤2 — ок** |
```

## Процесс разработки

### Spec-first

> Любая нетривиальная задача начинается со спеки. Код без спеки не пишем.

1. Написать спеку → `docs/internal/AS-NNN-название.md`
2. Оценить по 7 углам
3. Если >2 девяток или есть оценка <9 → доработать
4. Обсудить с пользователем
5. Только после этого — код

### Когда спека не нужна

- Тривиальная правка (опечатка, форматирование)
- Баг с очевидным фиксом
- Явно сказано «сделай как считаешь нужным»
